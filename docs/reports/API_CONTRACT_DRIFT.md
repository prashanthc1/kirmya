## Global / Cross-cutting

### G1 ŌĆö Success envelope shape disagrees across all three sources  *(High)*
- **Code** (`common/response.go` `WriteSuccess`): every success body is `{ "success": true, "data": <payload>, "meta": { "timestamp": "<RFC3339>" } }`.
- **Markdown doc** (┬¦1 Conventions): "Success envelope: `{ "data": ... }`" ŌĆö no `success`, no `meta`.
- **Frontend** (`api.ts` `Envelope<T>`): `{ success: boolean; data: T; error? }` ŌĆö has `success` but **no `meta`**.
- **Discrepancy:** three different envelope definitions. The frontend's `request()` reads only `body.data` / `body.error`, so it tolerates the extra `meta`, but the markdown contract is simply wrong and any new consumer trusting it would be surprised by `success`/`meta`.
- **Canonical:** `{ "success": true, "data": ..., "meta": { "timestamp": ... } }` (match code). Fix the markdown doc; optionally add `meta` to the TS `Envelope`.

### G2 ŌĆö Cursor pagination (`next_cursor`) is documented but not implemented  *(High)*
- **Markdown doc** ┬¦1: "Pagination: cursor-based ŌĆö `?limit=20&cursor=<opaque>`; response includes `{ "data": [...], "next_cursor": "..." }`."
- **Code/Frontend:** no `next_cursor`, no `cursor` query param anywhere. List endpoints return bare arrays under a named key (e.g. `{ "jobs": [...] }`, `{ "notifications": [...], "unread": N }`) or use `limit`/`offset` (admin users). The frontend never sends a `cursor`.
- **Discrepancy:** the documented pagination model does not exist.
- **Canonical:** either (a) document the real per-endpoint shapes (named array key; `limit`/`offset` for admin), or (b) implement cursor pagination. Recommend (a) ŌĆö correct the doc to reflect code, since no code uses cursors.
